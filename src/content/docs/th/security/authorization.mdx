ในบทเรียนที่แล้วเราจัดการเรื่อง *ใครอยู่บนการเชื่อมต่อ* บทเรียนนี้เป็นเรื่องคำถามที่ใหญ่กว่ามากซึ่งตามมา: *message ตัวนี้โดยเฉพาะได้รับอนุญาตไหม?* Authentication เกิดขึ้น **ครั้งเดียว**; authorization เกิดขึ้น **ทุก message** การรวมสองสิ่งนี้เข้าด้วยกันเป็นหนึ่งใน WebSocket bug ที่พบบ่อยที่สุดและแพงที่สุด

## Authenticate ครั้งเดียว authorize ทุก action

HTTP API ตรวจ permission ใหม่ในแต่ละ request แทบจะโดยบังเอิญ — แต่ละ request แยกกัน ดังนั้นแต่ละอันจึงวิ่งผ่าน middleware ของคุณ การเชื่อมต่อ WebSocket ทำลายนิสัยนั้น หลังจาก handshake message เป็นร้อยมาถึงบนการเชื่อมต่อที่ authenticate แล้ว **อันเดียว** และมันชวนให้คิดว่า "การเชื่อมต่อนี้ล็อกอินอยู่ ดังนั้น message ของมันก็โอเค"

มันไม่โอเค *ล็อกอินอยู่* ไม่ใช่ *ได้รับอนุญาต* ผู้ใช้ที่ authenticate เป็น `alice` อาจส่ง:

```json
{ "type": "delete", "room": "admins", "messageId": "42" }
```

`alice` จะลบ message นั้นได้ไหมไม่เกี่ยวกับว่า `alice` authenticate แล้วหรือไม่ — มันขึ้นอยู่กับ role ของ `alice`, การเป็นสมาชิกในห้องนั้น, และความสัมพันธ์ของเธอกับ message นั้นโดยเฉพาะ การตัดสินใจนั้นต้องทำใหม่ ฝั่ง server ตอนที่มันมาถึง

## อย่าไว้ใจตัวตนที่ client อ้าง

นี่คือบาปร้ายแรง และมันมีอยู่ทุกที่ message ข้างบนอาจมาถึงในรูปแบบนี้แทน:

```json
{ "type": "delete", "user": "admin", "room": "admins", "messageId": "42" }
```

server ที่ไร้เดียงสาอ่าน `msg.user` แล้วคิดว่า "อา admin ขอมา" แต่ client *เขียน field นั้นเอง* client อ้างเป็นใครก็ได้ ตัวตนที่เชื่อถือได้เพียงอย่างเดียวคือตัวตนที่ **server ของคุณ pin ไว้กับการเชื่อมต่อตอน handshake** — `ws.user` จากบทเรียนที่แล้ว — ซึ่งสร้างจาก credential ที่ validate แล้วและ client ปลอมแปลงไม่ได้

ดังนั้นกฎจึงตรงไปตรงมา: **เพิกเฉยต่อตัวตนใด ๆ ใน body ของ message** Authorize เทียบกับ `ws.user` ค่าที่คุณตั้ง ไม่ใช่เทียบกับ field `user`/`role`/`isAdmin` ที่ client ส่งมา ถ้า client เขียนมันได้ มันก็เป็นข้ออ้าง ไม่ใช่ข้อเท็จจริง

## เดโม authorization ราย message ที่รันได้จริง

เดโมด้านล่างคือ body ของ `ws` connection handler handshake ได้ pin ตัวตนไว้กับ socket แล้ว — ตรงนี้เรา hard-code `ws.user` และตาราง role เพื่อให้เดโมอยู่ครบในตัว จากนั้น **ทุก** message ที่เข้ามาจะถูก authorize: เรา look up role *จริง* ของ `ws.user` และเราจงใจ **เพิกเฉย** ต่อ field `user` ที่ client ส่งมา `delete` อนุญาตเฉพาะ admin เท่านั้น; `post` อนุญาตทุกคน เปิดมันใน StackBlitz แล้วให้ client ส่ง message ชนิดต่าง ๆ

export const authzServerCode = `// Body of wss.on('connection', (ws, req) => { ... }). ws/wss/req in scope.
// Handshake already authenticated this socket. Pin the REAL identity here:
ws.user = 'alice';                       // came from a validated credential
const ROLES = { alice: 'member', bob: 'admin' };

function can(user, action) {
  const role = ROLES[user] || 'guest';   // look up the REAL role server-side
  if (action === 'post') return true;    // anyone authenticated may post
  if (action === 'delete') return role === 'admin';
  return false;
}

ws.on('message', (data) => {
  let msg;
  try { msg = JSON.parse(data.toString()); }
  catch { ws.send(JSON.stringify({ type: 'error', reason: 'bad json' })); return; }

  // Authorize against ws.user — the pinned identity — NOT msg.user.
  // Even if the client sends { user: 'admin' }, we never read it.
  if (!can(ws.user, msg.type)) {
    console.log(\`DENY \${ws.user} -> \${msg.type} (claimed user: \${msg.user})\`);
    ws.send(JSON.stringify({ type: 'error', reason: 'forbidden', action: msg.type }));
    return;
  }

  console.log(\`ALLOW \${ws.user} -> \${msg.type}\`);
  ws.send(JSON.stringify({ type: 'ok', action: msg.type }));
});`;

เพราะ `ws.user` คือ `alice` ซึ่งเป็น `member`, `post` จึงได้รับอนุญาตแต่ `delete` ถูกปฏิเสธ — **แม้ว่า client จะเพิ่ม `"user": "admin"` ลงใน message** เพราะ server ไม่เคยอ่าน field นั้น สลับ `ws.user` เป็น `'bob'` แล้ว `delete` จะได้รับอนุญาต นั่นคือวินัยทั้งหมด: ตัวตนมาจากการเชื่อมต่อ, permission ถูกคำนวณใหม่ราย message, และ body ของ message ถูกมองเป็นคำขอ ไม่ใช่แหล่งของความจริง

| Authorization Check | เมื่อไร | ตัวอย่าง |
|---|---|---|
| Connection level | หลัง handshake | user ต้อง authenticate ก่อนส่ง message ใด ๆ |
| Room/Channel level | ก่อน join | user มีสิทธิ์ join channel นี้ไหม |
| Message level | ก่อน process | user มีสิทธิ์ส่ง message type นี้ไหม |
| Resource level | ก่อน action | user เป็นเจ้าของ document นี้ไหม |

## Common Mistakes

**Check Auth ที่ Connection เท่านั้น**
อาการ:
- validate token เมื่อ connect แต่ไม่ check permission ต่อ room หรือ action
- user authenticate แล้วสามารถ join ทุก room และส่ง message ทุกประเภท
- check permission ทุก action: join room, send message, subscribe channel

**Token ที่ Expire แต่ Connection ยังอยู่**
อาการ:
- user connect ด้วย JWT ที่ expire ใน 1 ชั่วโมง
- หลัง 1 ชั่วโมง connection ยังเปิดอยู่ — user ยังส่ง message ได้
- validate token expiry ต่อ message หรือ implement token refresh ผ่าน WebSocket

> 💡 Real World Example
>
> **Slack:**
> - authorization ต่อ channel — ไม่ใช่ทุก user เห็นทุก channel
> - message ที่ส่งไปยัง channel ที่ user ไม่มีสิทธิ์ → reject
>
> **GitHub Codespaces:**
> - authorization ต่อ codespace — เข้าถึงได้เฉพาะ owner และ collaborator
> - terminal WebSocket reject ถ้า token ไม่มีสิทธิ์ต่อ codespace นั้น

<Callout title="การเชื่อมต่อรู้ว่าคุณเป็นใคร message ไม่รู้">
Authenticate **ครั้งเดียว** ที่ handshake แล้ว pin ตัวตนไว้กับ socket Authorize **ทุก** message เทียบกับตัวตนที่ pin ไว้นั้น — ไม่ใช่เทียบกับ user หรือ role ที่ client ใส่ไว้ใน payload client อ้างเป็นใครก็ได้; มีเพียงตัวตนฝั่ง server เท่านั้นที่จริง
</Callout>
