Handshake เปิดการเชื่อมต่อให้ จากจุดนั้นเป็นต้นไป ทุกไบต์ของทราฟฟิก WebSocket เคลื่อนที่ในแพ็กเก็ตขนาดเล็กที่มีโครงสร้างซึ่งเรียกว่า **frame** คุณจะแทบไม่ได้จัดการ frame ด้วยมือ — `WebSocket` API ของเบราว์เซอร์และไลบรารีฝั่ง server ทำให้คุณ — แต่การรู้ว่าเกิดอะไรขึ้นภายใต้การเรียก `ws.send('hello')` ของคุณ ทำให้ส่วนที่เหลือของคอร์สเข้าที่เข้าทาง

## Frame คืออะไร

Frame คือหน่วยกะทัดรัดที่มี header เล็ก ๆ ตามด้วย payload header คือที่ที่ความฉลาดของโปรโตคอลอาศัยอยู่ มันแบกสิ่งต่าง ๆ เหล่านี้ ท่ามกลางสิ่งอื่น:

- **opcode** ที่บอกว่า frame นี้เป็น *ชนิด* ใด
- บิต **mask** และ (สำหรับ client frame) masking key
- ฟิลด์ **length** สำหรับ payload
- บิต **FIN** ที่บ่งบอกว่า frame นี้ทำให้ข้อความสมบูรณ์หรือไม่

"ข้อความ" เชิงตรรกะหนึ่งข้อความของคุณกลายเป็น frame หนึ่งอันหรือมากกว่าบนสาย

## Opcodes: ชนิดของ frame

opcode เพียงไม่กี่ตัวครอบคลุมทุกอย่างที่ WebSocket ทำ:

- **Text** — สตริง UTF-8 นี่คือสิ่งที่คุณได้จาก `ws.send('hello')`
- **Binary** — ไบต์ดิบ สำหรับสิ่งต่าง ๆ เช่น รูปภาพ ชิ้นส่วนเสียง หรือการเข้ารหัสที่มีประสิทธิภาพ
- **Ping** และ **Pong** — คู่ heartbeat ฝ่ายหนึ่งส่ง ping อีกฝ่ายมีหน้าที่ต้องตอบด้วย pong มันให้ปลายฝ่ายใดฝ่ายหนึ่งตรวจสอบได้ว่าการเชื่อมต่อยังมีชีวิตอยู่โดยไม่ต้องส่งข้อมูลแอปพลิเคชัน (เราจะใช้สิ่งเหล่านี้อย่างหนักเมื่อไปถึงโมดูลวงจรชีวิตของการเชื่อมต่อ)
- **Close** — การพูดอย่างสุภาพว่า "ฉันกำลังปิดสิ่งนี้" โดยอาจแบก status code และเหตุผลมาด้วย การปิดที่สะอาดคือทั้งสองฝ่ายแลกเปลี่ยน close frame กัน

## Masking: client ไป server

มีกฎหนึ่งข้อที่ทำให้มือใหม่ประหลาดใจ: **ทุก frame ที่ส่งจาก client ไป server ต้องถูก mask** client XOR payload ของมันกับ key สุ่มขนาด 4 ไบต์ที่มันรวมไว้ใน frame server unmask มันเมื่อมาถึง นี่ไม่ใช่การเข้ารหัส — ใครก็ตามที่อ่านไบต์สามารถย้อนกลับมันได้ — และ `wss://` คือสิ่งที่ปกป้องข้อมูลจริง ๆ Masking มีอยู่ด้วยเหตุผลที่แคบกว่า: เพื่อหยุดยั้งเพจที่มุ่งร้ายไม่ให้สร้าง WebSocket payload ที่ทำให้ HTTP proxy เก่า ๆ ที่ไม่รู้จัก WebSocket ซึ่งนั่งอยู่ระหว่าง client กับ server สับสน frame จาก server ไป client **ไม่ได้** ถูก mask

## Fragmentation

ข้อความขนาดใหญ่ไม่จำเป็นต้องถูกส่งเป็น frame ยักษ์อันเดียว มันสามารถถูก **fragment** ออกเป็นหลาย frame: frame แรกแบก opcode พร้อมบิต FIN ที่ไม่ถูกตั้ง frame กลาง ๆ ดำเนินต่อ payload และ frame สุดท้ายตั้ง FIN เพื่อทำเครื่องหมายจุดสิ้นสุด ผู้รับประกอบมันกลับเป็นข้อความเดียว สิ่งนี้ให้ผู้ส่งเริ่มสตรีมข้อความได้ก่อนที่จะรู้ความยาวทั้งหมด — มีประโยชน์สำหรับข้อมูลขนาดใหญ่หรือข้อมูลที่ถูกสร้างขึ้น

## ws:// เทียบกับ wss://

URL scheme สองแบบ สะท้อนเว็บเป๊ะ ๆ:

- **`ws://`** — การเชื่อมต่อ WebSocket ที่ไม่เข้ารหัส เหมาะสำหรับการพัฒนาในเครื่อง แต่ไม่ปลอดภัยเมื่อข้ามอินเทอร์เน็ตสาธารณะ เพราะใครก็ตามที่อยู่ตรงกลางสามารถอ่านและดัดแปลง frame ได้
- **`wss://`** — WebSocket บน TLS รูปแบบที่เข้ารหัส มันคือ WebSocket ที่เทียบเท่ากับ `https://` และเป็นค่าเริ่มต้นที่คุณควรเอื้อมไปหาในโปรดักชัน เดโมสดทุกอันในคอร์สนี้เชื่อมต่อด้วย `wss://`

## คุณแทบไม่ได้แตะต้องสิ่งเหล่านี้เลย

นี่คือส่วนที่ทำให้อุ่นใจ: API จัดการ frame, masking และการประกอบกลับให้คุณ เมื่อคุณเขียน `ws.send('hello, server')` ไลบรารีเข้ารหัส masked text frame เมื่อข้อความมาถึง มันส่งสตริงหรือ blob ที่ประกอบกลับแล้วอย่างสะอาดให้คุณใน `onmessage` คุณออกแบบ *ข้อความ* ส่วนโปรโตคอลเคลื่อนย้าย *frame* การรู้จัก frame layer เพียงแค่หมายความว่าคุณจะเข้าใจสิ่งที่เกิดขึ้นอย่างชัดเจนเมื่อเราปรับ heartbeat จัดการข้อมูล binary และดีบักการปิดในภายหลัง

| Frame Type | ใช้เมื่อ | ข้อควรระวัง |
|---|---|---|
| Text (UTF-8) | ส่ง JSON, string message | ขนาด payload ไม่ควรใหญ่เกิน 64KB ต่อ frame |
| Binary | ส่ง image, audio, Protobuf | ต้อง parse ที่ receiver อย่างระวัง |
| Ping/Pong | heartbeat ตรวจ connection มีชีวิต | browser ไม่ให้ JS ส่ง Ping เอง — เป็นหน้าที่ server |
| Close | ปิด connection อย่างนุ่มนวล | ควรรอ close frame ตอบกลับก่อน destroy socket |

## Common Mistakes

**ส่ง Payload ขนาดใหญ่โดยไม่ Fragment**
อาการ:
- ส่ง JSON ขนาด 10MB ใน single frame
- receiver ต้องรอ frame ทั้งหมดก่อน process — blocking
- ใช้ fragmentation หรือ pagination message แทน

**ใช้ `ws://` ใน Production**
อาการ:
- ข้อมูลวิ่งแบบ plaintext ผ่าน network
- ถูก intercept หรือ inject โดย MITM ได้
- ใช้ `wss://` เสมอใน production

> 💡 Real World Example
>
> **Google Meet:**
> - ใช้ Binary frame สำหรับ video/audio data (VP8, Opus codec)
> - ใช้ Text frame สำหรับ control message (mute, chat, participant list)
>
> **Figma:**
> - ส่ง cursor position และ selection state เป็น Binary frame (Protobuf)
> - ลด payload size เมื่อเทียบกับ JSON text frame ได้ถึง 60%

<Callout title="บทสรุปของ frame layer">
หลัง handshake ข้อมูลเดินทางในรูปของ **frame** ที่มี opcode สำหรับ text, binary, ping/pong และ close frame จาก client ไป server ถูก **mask** ข้อความสามารถถูก **fragment** ได้ และโปรดักชันใช้ **`wss://`** (TLS) `WebSocket` API ทำสิ่งเหล่านี้ทั้งหมดให้คุณ — คุณทำงานในระดับข้อความ ไม่ใช่ frame
</Callout>
