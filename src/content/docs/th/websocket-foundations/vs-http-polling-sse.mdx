WebSocket ไม่ใช่วิธีเดียวในการย้ายข้อมูลระหว่างเบราว์เซอร์กับเซิร์ฟเวอร์ และมันก็ไม่ได้เป็นวิธีที่ถูกต้องเสมอไป การจะเลือกได้ดี คุณต้องรู้จักทางเลือกอื่น ๆ บทเรียนนี้จะพาเดินผ่านสี่แนวทางอื่นและจัดมันเรียงเทียบกับ WebSocket บนสองแกนที่สำคัญที่สุด: **latency** (ไคลเอนต์รู้เรื่องเหตุการณ์ฝั่งเซิร์ฟเวอร์เร็วแค่ไหน) และ **overhead** (มีงานที่สูญเปล่ามากแค่ไหนต่อข้อความที่มีประโยชน์หนึ่งข้อความ)

## HTTP request/response ธรรมดา

เส้นฐาน ไคลเอนต์ถาม เซิร์ฟเวอร์ตอบ การแลกเปลี่ยนจบลง เซิร์ฟเวอร์ *ไม่มีทาง* พูดก่อนได้ สำหรับการดึงหน้าและส่งฟอร์ม สิ่งนี้สมบูรณ์แบบ แต่สำหรับการรู้เรื่องเหตุการณ์ที่เกิดบนเซิร์ฟเวอร์ มันไม่ได้ให้อะไรเลย — ไคลเอนต์จะต้องถามอีกครั้ง ซึ่งนำเราไปสู่ polling

## Short polling

ไคลเอนต์ถามว่า "มีอะไรใหม่ไหม?" ตามตัวจับเวลาคงที่ — สมมติทุกไม่กี่วินาที — และเซิร์ฟเวอร์ตอบทันที โดยส่วนใหญ่ตอบว่า "ไม่มี" เป็นเวลาส่วนใหญ่

**Latency** ถูกจำกัดด้วยช่วงเวลาของคุณ — เหตุการณ์หนึ่งอาจนั่งรออยู่โดยไม่มีใครเห็นได้นานถึงหนึ่งรอบ polling **Overhead** สูง: request ส่วนใหญ่แบก HTTP header เต็ม ๆ แต่ได้อะไรที่ไม่มีประโยชน์กลับมา กระชับช่วงเวลาให้แน่นขึ้นแล้วคุณตัด latency ลงได้ แต่ก็เพิ่ม request ที่สูญเปล่าเป็นทวีคูณ มันเรียบง่ายและทำงานได้ทุกที่ ซึ่งเป็นคุณงามความดีที่แท้จริงเพียงข้อเดียวของมัน

## Long polling

ตัวแปรที่ฉลาดกว่า ไคลเอนต์ส่ง request และเซิร์ฟเวอร์ **ถือมันเปิดค้างไว้** จนกว่าจะมีเหตุการณ์เกิดขึ้นหรือถึงเวลา timeout แล้วจึงตอบกลับ ไคลเอนต์ส่ง request อีกอันทันทีและรอต่ออีกครั้ง

วิธีนี้ดัน latency ลงใกล้เคียงเรียลไทม์ — response มาทันทีที่เหตุการณ์มา — โดยไม่ต้องมีตัวจับเวลาที่วุ่นวาย แต่ทุกข้อความที่ส่งได้ก็ยังเสียค่าใช้จ่ายเป็นรอบ request/response เต็ม ๆ และการเชื่อมต่อใหม่ ดังนั้นภายใต้อัตราเหตุการณ์ที่หนัก overhead จะไต่สูงขึ้น และมันสามารถบีบขีดจำกัดการเชื่อมต่อของเซิร์ฟเวอร์ได้

## Server-Sent Events (SSE)

SSE เปิดการเชื่อมต่อ HTTP ที่มีอายุยาว **เส้นเดียว** ซึ่งเซิร์ฟเวอร์สตรีมเหตุการณ์ไปยังไคลเอนต์ตามที่มันเกิดขึ้น มันมีประสิทธิภาพ เชื่อมต่อใหม่อัตโนมัติ และวิ่งบน HTTP ธรรมดา แต่มันถูกออกแบบให้เป็น **ทางเดียว: เซิร์ฟเวอร์ → ไคลเอนต์ เท่านั้น** ไคลเอนต์ส่งข้อมูลกลับผ่านช่องเดียวกันไม่ได้ การจะคุยกับเซิร์ฟเวอร์ มันต้องส่ง HTTP request ธรรมดาแยกต่างหาก

## WebSocket

WebSocket ก็เปิดการเชื่อมต่อเส้นเดียวค้างไว้เช่นกัน แต่มันเป็น **สองทิศทาง**: หลัง handshake ทั้งสองฝ่าย push ข้อความได้อย่างอิสระด้วย overhead ต่อข้อความที่น้อยที่สุด latency ต่ำเท่าที่เครือข่ายจะอนุญาตได้ใน *ทั้งสอง* ทิศทาง และไม่มีภาษี header ต่อข้อความ

## วางเทียบกัน

| แนวทาง | ทิศทาง | Latency ถึงไคลเอนต์ | Overhead ต่อข้อความ | เหมาะที่สุดเมื่อ |
| --- | --- | --- | --- | --- |
| HTTP request/response | ไคลเอนต์ → เซิร์ฟเวอร์ (ครั้งเดียว) | N/A (ไม่มี push) | หนึ่งรอบเต็มต่อ request | หน้าเพจ ฟอร์ม API ธรรมดา |
| Short polling | ไคลเอนต์ถามซ้ำ ๆ | สูงสุดหนึ่งช่วงเวลา | สูง (ส่วนใหญ่ว่างเปล่า) | การตรวจสอบเล็กน้อยเป็นครั้งคราว |
| Long polling | ไคลเอนต์ถาม เซิร์ฟเวอร์ถือไว้ | เกือบเรียลไทม์ | หนึ่งรอบต่อเหตุการณ์ | กึ่งเรียลไทม์บน HTTP ธรรมดา |
| SSE | เซิร์ฟเวอร์ → ไคลเอนต์ เท่านั้น | เกือบเรียลไทม์ | ต่ำ (หนึ่งสตรีม) | server push ไม่มี uplink จากไคลเอนต์ |
| WebSocket | ทั้งสองทิศทาง | ต่ำที่สุด ทั้งสองทาง | ต่ำที่สุด (ข้อความแบบ framed) | เรียลไทม์แบบโต้ตอบสองทาง |

## วิธีเลือก

- ถ้าไคลเอนต์เพียงแค่ **อ่าน** สตรีมของเหตุการณ์จากเซิร์ฟเวอร์เท่านั้น — ฟีดข่าว, การ tail log สด, อัปเดตความคืบหน้า — **SSE** เรียบง่ายกว่าและวิ่งบนโครงสร้างพื้นฐาน HTTP ธรรมดา
- ถ้าทั้งสองฝ่ายต้องส่งบ่อย ๆ และโต้ตอบกัน — แชต, การทำงานร่วมกัน, เกม, การควบคุมแบบสด — **WebSocket** คือตัวเลือกที่เป็นธรรมชาติ
- ถ้าคุณต้องการบางอย่างวันนี้บนโครงสร้างพื้นฐานที่พื้นฐานที่สุดและทราฟฟิกเบา **long polling** เป็นทางออกชั่วคราวที่สมเหตุสมผล ส่วน **short polling** ใช้เฉพาะกรณีที่เล็กน้อยจริง ๆ เท่านั้น

<Callout title="ความแตกต่างที่ทำให้คนสะดุด">
**SSE เป็นเซิร์ฟเวอร์ → ไคลเอนต์ เท่านั้น** **WebSocket เป็นสองทิศทาง** ถ้าไคลเอนต์ของคุณไม่จำเป็นต้อง push ผ่านช่องเดียวกันเลย SSE อาจเป็นทั้งหมดที่คุณต้องการ แต่เมื่อใดที่คุณต้องการทราฟฟิกสองทางอย่างแท้จริง ให้เอื้อมไปหา WebSocket
</Callout>
