WebSocket ไม่ได้โผล่ขึ้นมาจากอากาศ มันเริ่มต้นชีวิตในฐานะ HTTP request ที่ธรรมดาสมบูรณ์แบบ แล้วจึงถูก *upgrade* ในที่ การเข้าใจ handshake นี้คลายความลึกลับได้มากมาย: ทำไม WebSocket ถึงเดินทางบนพอร์ตเดียวกันกับทราฟฟิกเว็บ ทำไมมันถึงผ่าน proxy และ firewall ส่วนใหญ่ได้ และทำไม scheme `ws://` และ `wss://` ถึงสะท้อน `http://` และ `https://`

## มันเริ่มต้นในฐานะ HTTP

ไคลเอนต์เปิด TCP connection ธรรมดาและส่ง HTTP `GET` request — แต่พร้อม header พิเศษที่บอกว่า "ฉันอยากสลับการเชื่อมต่อนี้ไปเป็นโปรโตคอล WebSocket" สอง header ที่ทำงานจริงคือ `Upgrade: websocket` และ `Sec-WebSocket-Key`

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://example.com
```

มีบางตัวในนี้ที่สมควรกับตำแหน่งของมัน:

- **`Upgrade: websocket`** และ **`Connection: Upgrade`** ทำงานร่วมกันเพื่อส่งสัญญาณการสลับโปรโตคอล
- **`Sec-WebSocket-Key`** เป็นค่าสุ่มที่เข้ารหัส base64 ซึ่งไคลเอนต์สร้างขึ้นใหม่สำหรับการเชื่อมต่อนี้ มัน *ไม่ใช่* ความลับด้านความปลอดภัย มันมีอยู่เพื่อให้ไคลเอนต์ยืนยันได้ว่า response มาจากเซิร์ฟเวอร์ที่เข้าใจ WebSocket handshake อย่างแท้จริง
- **`Sec-WebSocket-Version: 13`** ตรึงเวอร์ชันของโปรโตคอลไว้ (13 คือเวอร์ชันมาตรฐาน)

## Response แบบ 101

ถ้าเซิร์ฟเวอร์พูด WebSocket ได้และยอมรับ มันจะตอบกลับด้วยสถานะ **`101 Switching Protocols`** — สถานะ HTTP ที่มีความหมายตรงตัวว่า "เราเสร็จสิ้นกับ semantics ของ HTTP บนการเชื่อมต่อนี้แล้ว จากนี้ไปมันเป็นโปรโตคอลที่ต่างออกไป"

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

ความมหัศจรรย์อยู่ที่ **`Sec-WebSocket-Accept`** เซิร์ฟเวอร์นำ `Sec-WebSocket-Key` ของไคลเอนต์มา ต่อท้ายด้วยสตริงคงที่ที่รู้จักกันดีซึ่งกำหนดโดย spec แฮชผลลัพธ์ด้วย SHA-1 แล้วเข้ารหัส base64 ไคลเอนต์คำนวณค่าเดียวกันและตรวจสอบว่าตรงกันไหม ถ้าตรงกัน ไคลเอนต์ก็รู้ว่ามันกำลังคุยกับเซิร์ฟเวอร์ WebSocket จริง ๆ ไม่ใช่ เช่น cache ที่คืน `101` กลับมาแบบมั่ว ๆ ไม่ตรงกัน ก็ไม่มีการเชื่อมต่อ

สังเกตสิ่งที่ไดอะแกรม *ไม่ได้* แสดง: การเชื่อมต่อเส้นที่สอง หลัง `101` ไคลเอนต์และเซิร์ฟเวอร์ยังคงใช้ **TCP connection เดิม** ที่มีอยู่แล้วต่อไป ไม่มีอะไรถูกเปิดใหม่ HTTP request เป็นเพียงประตูทางเข้า เมื่อผ่านมันไปแล้ว ทั้งสองฝ่ายหยุดแลกเปลี่ยนข้อความ HTTP และเริ่มแลกเปลี่ยน WebSocket frame (หัวข้อของบทเรียนถัดไป)

## ทำไมการออกแบบนี้ถึงฉลาด

เพราะ handshake *คือ* HTTP request การเชื่อมต่อ WebSocket จึงสามารถวิ่งบนพอร์ต 80 และ 443 เคียงข้างทราฟฟิกเว็บปกติ นำ TLS ที่มีอยู่มาใช้ซ้ำสำหรับ `wss://` และลอดผ่าน proxy และ load balancer ที่เข้าใจ HTTP อยู่แล้ว โปรโตคอลได้ระยะเอื้อมของเว็บมาฟรี ๆ แล้วจึงสลัด request/response model ของ HTTP ทิ้งทันทีที่ไม่ต้องการมันอีกต่อไป

<Callout title="สิ่งที่ต้องจำ">
WebSocket **เริ่มต้นในฐานะ HTTP GET** ที่แบก `Upgrade: websocket` และเซิร์ฟเวอร์ตอบ `101 Switching Protocols` หลังจากนั้น TCP connection เดิมเป๊ะ ๆ จะถูกนำกลับมาใช้ซ้ำ — ไม่มีการเชื่อมต่อใหม่ — เพื่อขนส่ง WebSocket frame ในทั้งสองทิศทาง
</Callout>
