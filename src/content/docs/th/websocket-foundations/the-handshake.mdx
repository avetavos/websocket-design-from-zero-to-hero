WebSocket ไม่ได้โผล่ขึ้นมาจากอากาศ มันเริ่มต้นชีวิตในฐานะ HTTP request ที่ธรรมดาสมบูรณ์แบบ แล้วจึงถูก *upgrade* ในที่ การเข้าใจ handshake นี้คลายความลึกลับได้มากมาย: ทำไม WebSocket ถึงเดินทางบนพอร์ตเดียวกันกับทราฟฟิกเว็บ ทำไมมันถึงผ่าน proxy และ firewall ส่วนใหญ่ได้ และทำไม scheme `ws://` และ `wss://` ถึงสะท้อน `http://` และ `https://`

## มันเริ่มต้นในฐานะ HTTP

client เปิด TCP connection ธรรมดาและส่ง HTTP `GET` request — แต่พร้อม header พิเศษที่บอกว่า "ฉันอยากสลับการเชื่อมต่อนี้ไปเป็นโปรโตคอล WebSocket" สอง header ที่ทำงานจริงคือ `Upgrade: websocket` และ `Sec-WebSocket-Key`

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://example.com
```

มีบางตัวในนี้ที่สมควรกับตำแหน่งของมัน:

- **`Upgrade: websocket`** และ **`Connection: Upgrade`** ทำงานร่วมกันเพื่อส่งสัญญาณการสลับโปรโตคอล
- **`Sec-WebSocket-Key`** เป็นค่าสุ่มที่เข้ารหัส base64 ซึ่ง client สร้างขึ้นใหม่สำหรับการเชื่อมต่อนี้ มัน *ไม่ใช่* ความลับด้านความปลอดภัย มันมีอยู่เพื่อให้ client ยืนยันได้ว่า response มาจาก server ที่เข้าใจ WebSocket handshake อย่างแท้จริง
- **`Sec-WebSocket-Version: 13`** ตรึงเวอร์ชันของโปรโตคอลไว้ (13 คือเวอร์ชันมาตรฐาน)

## Response แบบ 101

ถ้า server พูด WebSocket ได้และยอมรับ มันจะตอบกลับด้วยสถานะ **`101 Switching Protocols`** — สถานะ HTTP ที่มีความหมายตรงตัวว่า "เราเสร็จสิ้นกับ semantics ของ HTTP บนการเชื่อมต่อนี้แล้ว จากนี้ไปมันเป็นโปรโตคอลที่ต่างออกไป"

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

ความมหัศจรรย์อยู่ที่ **`Sec-WebSocket-Accept`** server นำ `Sec-WebSocket-Key` ของ client มา ต่อท้ายด้วยสตริงคงที่ที่รู้จักกันดีซึ่งกำหนดโดย spec แฮชผลลัพธ์ด้วย SHA-1 แล้วเข้ารหัส base64 client คำนวณค่าเดียวกันและตรวจสอบว่าตรงกันไหม ถ้าตรงกัน client ก็รู้ว่ามันกำลังคุยกับ server WebSocket จริง ๆ ไม่ใช่ เช่น cache ที่คืน `101` กลับมาแบบมั่ว ๆ ไม่ตรงกัน ก็ไม่มีการเชื่อมต่อ

## ทำไมการออกแบบนี้ถึงฉลาด

เพราะ handshake *คือ* HTTP request การเชื่อมต่อ WebSocket จึงสามารถวิ่งบนพอร์ต 80 และ 443 เคียงข้างทราฟฟิกเว็บปกติ นำ TLS ที่มีอยู่มาใช้ซ้ำสำหรับ `wss://` และลอดผ่าน proxy และ load balancer ที่เข้าใจ HTTP อยู่แล้ว โปรโตคอลได้ระยะเอื้อมของเว็บมาฟรี ๆ แล้วจึงสลัด request/response model ของ HTTP ทิ้งทันทีที่ไม่ต้องการมันอีกต่อไป

| Benefits (WebSocket Handshake) | Trade-offs |
|---|---|
| ใช้ HTTP infrastructure เดิม — ผ่าน firewall และ proxy ได้ | handshake round-trip เพิ่ม latency ตอนเปิด connection ครั้งแรก |
| วิ่งบน port 80/443 ร่วมกับ HTTP ธรรมดา | proxy เก่าที่ไม่รู้จัก WebSocket อาจ block Upgrade header |
| `Sec-WebSocket-Accept` ป้องกัน cache ที่ไม่เข้าใจ WS | ต้องการ server-side support — ไม่ใช่ทุก server ที่ support |
| TLS (`wss://`) ต่อยอดจาก HTTPS ที่มีอยู่แล้ว | HTTP/2 multiplexing ไม่ทำงานกับ WebSocket — ต้องใช้ HTTP/1.1 |

## Common Mistakes

**ไม่ Validate Origin ใน Handshake**
อาการ:
- ยอมรับทุก WebSocket connection โดยไม่ตรวจ `Origin` header
- เปิดช่องให้ cross-site WebSocket hijacking (CSWSH)
- ตรวจสอบ `Origin` header เทียบกับ whitelist ก่อน respond 101

**ทำ Auth หลัง Connection เปิดแล้ว**
อาการ:
- รับ handshake ก่อน ค่อย auth ทีหลัง ทำให้ connection เปิดอยู่ชั่วคราวโดยไม่ตรวจสอบ
- attach auth token ใน handshake (query param หรือ header) และ validate ก่อน respond 101

> 💡 Real World Example
>
> **Binance WebSocket API:**
> - ใช้ WSS handshake บน `wss://stream.binance.com:9443`
> - validate API key ใน handshake ก่อน accept connection
> - reject ด้วย HTTP 401 แทนการรับแล้ว close ทีหลัง
>
> **GitHub Copilot:**
> - handshake ผ่าน HTTPS/WSS ที่ port 443
> - ใช้ corporate proxy ที่รู้จัก Upgrade header ได้
