นักพัฒนาที่มาจาก HTTP แบกข้อสมมติที่ฝังลึกมาด้วย: *ฉันเรียกเอง response นี้ก็เลยเป็นของฉัน* ด้วย `fetch` คำตอบที่คุณ `await` คือคำตอบของคำถามที่คุณถามอย่างไม่กำกวม WebSocket ถอดการรับประกันนั้นออกไปเงียบ ๆ — และการได้มันคืนมาคือหนึ่งในแพตเทิร์นที่มีประโยชน์ที่สุดในการออกแบบ message

## socket เน้น message ไม่ใช่ request/reply

WebSocket ไม่มีแนวคิดของ request ที่จับคู่กับ response มีเพียงสตรีมของ message ที่ไหลไปแต่ละทิศทาง *อย่างเป็นอิสระ* เมื่อ `onmessage` ทำงาน message นั้นอาจเป็น:

- reply ของสิ่งที่คุณเพิ่งถาม,
- reply ของสิ่งที่คุณถามไปสาม message ก่อนหน้าซึ่งใช้เวลานานกว่า,
- หรือ server push ที่ไม่ได้ร้องขอเลย (chat line ใหม่, price tick)

ถ้าคุณยิง request "fetch user" ออกไปสามครั้งและผลลัพธ์กลับมาสามอัน **ไม่มีอะไรใน transport บอกคุณว่าผลลัพธ์ไหนตอบ request ไหน** reply อาจมาถึงผิดลำดับ สลับกับ push คุณพึ่งพาลำดับไม่ได้ และคุณ `await` `send` ไม่ได้แน่นอน

## correlation id ผูก reply เข้ากับ request ของมัน

วิธีแก้คือ field เดียวที่คุณรู้วิธีเพิ่มอยู่แล้ว: **`id`** ที่ไม่ซ้ำกันบนแต่ละ request ซึ่ง server **สะท้อนกลับ (echoes)** บน reply ที่จับคู่กัน client เก็บตารางเล็ก ๆ ของ *id → "ใครกำลังรออันนี้"* และเมื่อ reply มาถึง มันค้นหา id, resolve ผู้รอนั้น แล้วลบ entry ออก reply ของ id ที่ไม่รู้จัก (หรือ message ที่ไม่มี id เลย) ถูกถือว่าเป็น server push

## ชั้น request/reply เล็ก ๆ แบบสด ๆ

เดโมด้านล่างทำ `request()` บน echo socket ภายในหน้า แต่ละครั้งที่เรียกจะสร้าง `id` ใหม่ เก็บ resolver ไว้ใน map `pending` แล้วส่ง request เมื่อ reply กลับมา `id` ที่สะท้อนกลับของมันจะหา resolver ที่ถูกต้องเจอ request สองอันถูกส่งออกไป echo หน่วงเวลาต่างกันเพื่อให้ **reply มาถึงผิดลำดับ** — และ correlation ยังคงจับคู่แต่ละอันได้อย่างถูกต้อง มันใช้ `WebSocket` API จริง

export const reqReplyCode = `(function () {
  // In-page echo socket with the real WebSocket API. It replies to a 'getUser' request,
  // echoing the request id, with a small random delay so replies can arrive out of order.
  class EchoSocket {
    constructor() {
      this.readyState = 0;
      setTimeout(() => { this.readyState = 1; if (this.onopen) this.onopen({}); }, 50);
    }
    send(data) {
      const req = JSON.parse(data);
      const ages = { alice: 30, bob: 41 };
      const delay = req.name === 'alice' ? 120 : 30; // alice answers SLOWER than bob
      setTimeout(() => {
        if (this.onmessage) {
          const reply = { type: 'reply', id: req.id, name: req.name, age: ages[req.name] };
          this.onmessage({ data: JSON.stringify(reply) });
        }
      }, delay);
    }
    close(code) { this.readyState = 3; setTimeout(() => { if (this.onclose) this.onclose({ code: code || 1000 }); }, 10); }
  }

  const ws = new EchoSocket();
  const pending = {};      // id -> resolver function
  let nextId = 0;
  let answered = 0;

  // request() returns a promise that resolves when the reply with our id arrives.
  function request(payload) {
    return new Promise((resolve) => {
      const id = 'r' + (++nextId);
      pending[id] = resolve;
      ws.send(JSON.stringify(Object.assign({ id: id }, payload)));
    });
  }

  ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    const resolve = pending[msg.id];
    if (resolve) {
      delete pending[msg.id];   // one-time use
      resolve(msg);
    } else {
      console.log('unsolicited server push:', msg.type); // no matching id = a push
    }
  };

  function done() {
    answered += 1;
    if (answered === 2) ws.close();
  }

  ws.onopen = () => {
    console.log('asking for alice (slow) and bob (fast)...');
    request({ type: 'getUser', name: 'alice' }).then((r) => {
      console.log('alice resolved correctly, age', r.age, '(id ' + r.id + ')');
      done();
    });
    request({ type: 'getUser', name: 'bob' }).then((r) => {
      console.log('bob resolved correctly, age', r.age, '(id ' + r.id + ')');
      done();
    });
  };

  ws.onclose = () => console.log('both promises resolved to the RIGHT user despite out-of-order replies');
})();`;

ดูลำดับให้ดี: reply ของ bob กลับมาก่อนเพราะ server ตอบเขาเร็วกว่า แต่ promise ของ alice ยังคง resolve เป็น alice และของ bob เป็น bob `id` — ไม่ใช่ลำดับการมาถึง — คือสิ่งที่ทำให้การจับคู่ถูกต้อง map `pending` เล็ก ๆ นั้นคือกลเม็ดทั้งหมดเบื้องหลังทุก library "RPC over WebSocket" ที่คุณจะได้ใช้

ส่วนขัดเกลาสองสามอย่างที่คุณจะเพิ่มใน production: **timeout** ที่ reject entry ที่รออยู่หากไม่มี reply มาถึง และการเคลียร์ map `pending` หาก socket ปิด เพื่อไม่ให้ promise ค้างอยู่ตลอดกาล

| Pattern | ใช้เมื่อ | ข้อเสีย |
|---|---|---|
| Correlation ID + pending map | ต้องการ request/response บน WS | ต้อง implement เอง |
| HTTP สำหรับ request/response | query ครั้งเดียว, cacheable | latency สูงกว่า WS |
| Fire-and-forget (ไม่มี reply) | log, metric push | ไม่รู้ว่า server ได้รับหรือไม่ |

## Common Mistakes

**ไม่มี Timeout สำหรับ Pending Request**
อาการ:
- ส่ง request แล้ว server ไม่ตอบ — promise ค้างตลอดกาล
- memory leak จาก `pending` map ที่โตขึ้นเรื่อย ๆ
- ตั้ง timeout ต่อ request: `setTimeout(() => reject('timeout'), 5000)` แล้วลบออกจาก map

**ไม่ Clear Pending Map เมื่อ Connection Close**
อาการ:
- WS close → reconnect แต่ promise จาก session เก่ายังค้างอยู่
- memory leak และ stale handler ทำงานเมื่อ reconnect
- ใน `onclose`: reject + clear pending map ทั้งหมด

> 💡 Real World Example
>
> **Pusher Channels:**
> - ส่ง subscription request ด้วย `channel_id`
> - server ตอบ `pusher_internal:subscription_succeeded` พร้อม `channel` ที่ match
>
> **Phoenix LiveView (Elixir):**
> - ใช้ ref field เป็น correlation ID สำหรับทุก push/reply
> - client track ref เพื่อ match response กับ request ที่ส่งไป
