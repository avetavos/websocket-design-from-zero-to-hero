ในบทเรียนที่แล้วเราบอกว่าการไถลเงียบ ๆ จาก *live* ไปสู่ *dropped* คือ transition ที่แอปลืม บทเรียนนี้อธิบายว่า **ทำไมมันถึงเงียบ** — และมอบเครื่องมือหนึ่งเดียวที่ทำลายความเงียบนั้น: heartbeat

## Connection ที่เงียบและ connection ที่ตายดูเหมือนกัน

นี่คือความจริงที่อึดอัดเกี่ยวกับ socket ที่เปิดอยู่: **การไม่ทำอะไรเลยดูเหมือนการพังเป๊ะ ๆ** ถ้าไม่มีฝ่ายใดส่งข้อความเลยเป็นเวลาสามสิบวินาที จะมีคำอธิบายที่เป็นไปได้สองอย่าง และจากภายในแล้วมันแยกแยะกันไม่ออก:

- ทุกอย่างปกติดี และก็แค่ไม่มีใครมีอะไรจะพูด
- Connection ตายไปเมื่อสิบวินาทีที่แล้ว — Wi-Fi handoff แลปท็อปที่หลับ load balancer ที่กำจัด stream ที่ idle ทิ้ง — และ packet ที่ประกาศการตายของมันไม่เคยมาถึง

TCP เองก็ช่วยคุณไม่ได้ตรงนี้ TCP รู้ว่า connection พังก็ต่อเมื่อมันพยายามส่งแล้ว acknowledgement ไม่เคยกลับมา ถ้าแอปของคุณ *idle* TCP ก็ idle ด้วย ดังนั้นมันจึงไม่เคยพยายามส่ง จึงไม่เคยรู้ TCP keepalive ที่ติดมากับ operating system อาจใช้เวลาเป็น **ชั่วโมง** กว่าจะทำงานตามค่าเริ่มต้น — ไร้ประโยชน์สำหรับแอป real-time ผลก็คือ: object `WebSocket` ของคุณรายงานอย่างมีความสุขว่า `readyState === 1` (OPEN) บน connection ที่กลายเป็นซากปรักหักพังมาหลายนาทีแล้ว

ทางรักษาเดียวคือ **สร้าง traffic อย่างจงใจ** ในช่วงเวลาที่รู้ และตรวจสอบว่ามันกลับมา นั่นคือ heartbeat

## Heartbeat สองชนิด

มีกลไกที่แตกต่างกันสองอย่าง และผู้คนสับสนระหว่างสองอย่างนี้อยู่เสมอ

- **Protocol ping/pong frame** WebSocket protocol นิยาม control frame พิเศษไว้: *ping* frame และ *pong* frame ตามสเปก เมื่อ endpoint รับ ping มันต้องตอบด้วย pong สิ่งเหล่านี้อยู่ใต้ message handler ของคุณ — มันเป็นส่วนหนึ่งของ framing layer ที่ครอบคลุมในโมดูล foundations
- **Application-level heartbeat** ข้อความธรรมดาที่codeของคุณส่งและตอบ เช่น JSON `{"type":"ping"}` ที่ server ของคุณตอบกลับด้วย `{"type":"pong"}`

ทำไมถึงต้องสร้างเวอร์ชัน app-level ในเมื่อ protocol ให้ ping/pong มาฟรี ๆ อยู่แล้ว? เพราะ **เบราว์เซอร์ไม่อนุญาตให้คุณใช้ protocol frame เหล่านั้น** `WebSocket` API ของเบราว์เซอร์ไม่มี method `ping()` และไม่ยิง event ใดเมื่อ ping หรือ pong มาถึง — มันถูกจัดการอย่างมองไม่เห็นภายในเบราว์เซอร์ *server* ที่เป็นเบราว์เซอร์สามารถส่ง protocol ping ไปยังเบราว์เซอร์ได้ แต่code *client* ที่เป็นเบราว์เซอร์ไม่สามารถส่งหรือสังเกตมันได้ ดังนั้นในทางปฏิบัติ:

- **Server ไปยังเบราว์เซอร์:** server อาจใช้ protocol ping/pong จริงเพื่อตรวจสอบเบราว์เซอร์
- **เบราว์เซอร์ไปยัง server:** client ต้องใช้ข้อความ app-level เพราะนั่นคือทั้งหมดที่มันมองเห็น

ดังนั้นระบบ production ส่วนใหญ่จึงรัน heartbeat แบบ **app-level** เพื่อให้ logic เดียวกันทำงานได้บนทุกแพลตฟอร์ม นั่นคือสิ่งที่เราจะสร้างด้านล่าง

## The heartbeat loop

รูปแบบของ app-level heartbeat มักจะเหมือนกันเสมอ:

  end`;

ในcode มันคือ interval timer บวกกับ state ชิ้นเล็ก ๆ: ส่ง ping ทุก N วินาที และจำว่า pong ที่ตรงกันกลับมาหรือยัง นี่คือหัวใจของมันในรูปแบบ TypeScript ธรรมดา:

```ts
const HEARTBEAT_MS = 5000;

function startHeartbeat(ws: WebSocket) {
  const timer = setInterval(() => {
    if (ws.readyState !== WebSocket.OPEN) return;
    ws.send(JSON.stringify({ type: 'ping', t: Date.now() }));
  }, HEARTBEAT_MS);

  ws.addEventListener('message', (event) => {
    const msg = JSON.parse(event.data as string);
    if (msg.type === 'pong') {
      // The connection answered — it is genuinely alive right now.
    }
  });

  ws.addEventListener('close', () => clearInterval(timer));
}
```

การส่ง ping คือครึ่งที่ง่าย ครึ่งที่มีค่า — *การสังเกตว่า pong ไม่เคยมา* — คือบทเรียนถัดไปทั้งบท สำหรับตอนนี้ ให้โฟกัสที่จังหวะ: ชีพจรที่สม่ำเสมอ และคำตอบสำหรับทุกครั้งที่เต้น

## ping/pong แบบสด

เดโมด้านล่างใช้ socket ในหน้าที่มี `WebSocket` API จริง mock เล่นบทเป็น server ที่ประพฤติดี: ทุกครั้งที่มันรับข้อความ `ping` แบบ app-level มันจะตอบด้วย `pong` client ส่ง ping ทุกหนึ่งวินาทีและ log การไปกลับ ดู heartbeat ลงตัวเข้าสู่จังหวะของมัน

export const pingPongCode = `(function () {
  // In-page socket with the real WebSocket API. The mock acts as a healthy
  // server: it answers every app-level {type:'ping'} with a {type:'pong'}.
  // In a real app: const ws = new WebSocket('wss://your-server').
  class HeartbeatSocket {
    constructor() {
      this.readyState = 0;
      setTimeout(() => { this.readyState = 1; if (this.onopen) this.onopen({}); }, 30);
    }
    send(data) {
      const msg = JSON.parse(data);
      if (msg.type === 'ping') {
        // Healthy server: reply with a pong echoing the sequence number.
        setTimeout(() => {
          if (this.onmessage) this.onmessage({ data: JSON.stringify({ type: 'pong', seq: msg.seq }) });
        }, 20);
      }
    }
    close(code) { this.readyState = 3; if (this.onclose) this.onclose({ code: code || 1000 }); }
  }

  const ws = new HeartbeatSocket();
  let seq = 0;
  let timer = null;
  ws.onopen = () => {
    console.log('open — starting heartbeat (1 ping/sec)');
    timer = setInterval(() => {
      if (ws.readyState !== 1) return;
      seq += 1;
      console.log('-> ping #' + seq);
      ws.send(JSON.stringify({ type: 'ping', seq: seq }));
    }, 1000);
  };
  ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    if (msg.type === 'pong') {
      console.log('<- pong #' + msg.seq + ' — connection is alive');
      if (msg.seq === 3) { clearInterval(timer); console.log('demo done — stopping heartbeat'); ws.close(); }
    }
  };
  ws.onclose = (event) => console.log('closed, code', event.code);
})();`;

คุณจะเห็น ping ออกไปและ pong กลับมา หนึ่งคู่ต่อวินาที ทุก pong ที่กลับมาคือหลักฐานชัดเจนว่า connection ยังมีชีวิต *ในขณะนั้น* — ไม่ใช่เมื่อสามสิบวินาทีที่แล้ว ไม่ใช่ "น่าจะ" นั่นคือจุดประสงค์ทั้งหมดของ heartbeat: มันเปลี่ยนความเงียบ ซึ่งไม่บอกอะไรคุณเลย ให้กลายเป็นสัญญาณที่สม่ำเสมอซึ่งคุณเชื่อถือได้จริง

| Heartbeat Type | ทำงานอย่างไร | ข้อดี |
|---|---|---|
| Protocol-level Ping/Pong | WebSocket frame opcode 0x9/0xA | browser handle อัตโนมัติ, lightweight |
| Application-level Ping | ส่ง JSON `{ "type": "ping" }` | flexible, บน binary หรือ text frame |
| Keepalive TCP | OS-level TCP keepalive | ไม่ต้องใช้ application code |

## Common Mistakes

**ไม่มี Heartbeat เลย**
อาการ:
- connection ดูเหมือนเปิดอยู่ใน server แต่ client จริงหายไป (zombie connection)
- load balancer หรือ firewall ปิด idle connection โดยไม่แจ้ง
- implement server-side ping ทุก 30 วินาที, ถ้าไม่มี pong → ปิด connection

**Heartbeat Interval สั้นเกิน**
อาการ:
- ping ทุก 1 วินาท — server และ client เสียเวลา ping/pong แทนที่จะส่งข้อมูล
- battery drain บน mobile device
- interval 30-60 วินาทีเหมาะสำหรับส่วนใหญ่ — ขึ้นกับ firewall timeout

> 💡 Real World Example
>
> **Discord Gateway:**
> - ส่ง `{ "op": 1, "d": null }` (Heartbeat opcode) ตาม interval ที่ server กำหนด
> - server ตอบ `{ "op": 11 }` (Heartbeat ACK) — client รู้ว่า connection alive
>
> **Slack RTM API:**
> - ping/pong ทุก 30 วินาที
> - ถ้าไม่มี pong ใน 10 วินาที → reconnect อัตโนมัติ
