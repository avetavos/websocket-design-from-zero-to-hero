จนถึงตอนนี้คอร์สนี้อยู่ฝั่ง **client** เป็นหลัก — เปิด socket, ส่ง frame, เฝ้าดูข้อความวิ่งเข้ามา โมดูลนี้จะหันกล้องกลับด้าน WebSocket server คือ process ที่รันค้างไว้นาน คอย holding connection ที่เปิดอยู่ **จำนวนมาก** พร้อมกัน และตัดสินใจว่าใครจะได้ยินอะไร นั่นเป็น mental model ที่ต่างจาก HTTP handler อย่างแท้จริง และการทำให้ถูกต้องคือเส้นแบ่งระหว่างเดโมกับระบบจริง

## WebSocket server ทำอะไรกันแน่

HTTP server คิดในแง่หนึ่ง request หนึ่ง response แล้วก็ลืมคุณไป WebSocket server ลืมคุณไม่ได้ — คุณยังอยู่บนสายอยู่ ถ้าถอดงานออกมาเป็นแก่น มันมีความรับผิดชอบจริง ๆ อยู่สี่อย่าง:

- **รับ upgrade (Accept upgrades).** connection เริ่มต้นชีวิตในฐานะ HTTP request ธรรมดาที่พก header `Upgrade: websocket` มาด้วย server ตรวจสอบความถูกต้องแล้วสลับ socket ไปใช้โปรโตคอล WebSocket จากจุดนั้นเป็นต้นไปมันคือท่อแบบถาวร ไม่ใช่ request อีกต่อไป
- **ติดตาม connection (Track connections).** ทุก socket ที่รับเข้ามาต้องถูกจดจำ server เก็บชุดของผู้ที่เชื่อมต่ออยู่แบบ live ไว้เพื่อให้หากลับมาได้ในภายหลัง — เพื่อ push update, เพื่อนับจำนวน, หรือเพื่อ clean up เมื่อพวกเขาจากไป
- **Route ข้อความ (Route messages).** เมื่อ frame เข้ามา server อ่านมันแล้วตัดสินใจว่ามันหมายถึงอะไร: บรรทัดแชทสำหรับ room หนึ่ง, คำขอ subscribe, ping, หรือคำสั่ง การ routing เปลี่ยน byte ดิบให้กลายเป็นพฤติกรรมของแอปพลิเคชัน
- **Broadcast.** ลูกเล่นเด่น เพราะ server holding ทุก socket ที่เปิดอยู่ มันจึงส่ง event เดียวไปยัง client *จำนวนมาก* ได้พร้อมกัน — ในชั่วขณะที่มันเกิดขึ้น โดยที่ไม่มีใครต้องร้องขอ

สังเกตว่าสามในสี่อย่างนี้เป็นสิ่งที่ HTTP server ธรรมดาไม่เคยต้องทำ การ tracking, การ routing บน connection ที่ถาวร และการ fan-out จะมีความหมายก็ต่อเมื่อ connection **ยังเปิดค้างอยู่**

## หนึ่ง server, หลาย client

วิธีที่ดีในการมองภาพ WebSocket server คือมองเป็น hub ที่ถือสายหนึ่งเส้นไปยังแต่ละ client ข้อความจาก client เดี่ยว ๆ คนใดก็ตามสามารถเข้ามา ถูกประมวลผล แล้วก็ส่งกลับออกไปยัง client หนึ่งราย, กลุ่มย่อย, หรือทุกคน:

## โมดูลนี้ครอบคลุมอะไร

โมดูลนี้ — **การออกแบบ Server** — สร้างความรู้ WebSocket ฝั่ง server ครึ่งหนึ่งของคุณ ทีละชั้น:

1. **การออกแบบ Server (คุณอยู่ที่นี่)** — งานสี่อย่างของ server และ `ws` server ตัวแรกที่รันได้จริง
2. **A WebSocket server** — ไลบรารี `ws`: `WebSocketServer`, `connection` event, `ws.on('message')`, และ `ws.send`
3. **Connection state** — การแนบข้อมูลต่อ socket (user id, subscriptions) และ clean up มันตอน close
4. **Rooms and broadcast** — การจัดกลุ่ม client เข้าเป็น channel และ fan ข้อความออกไปยัง room หนึ่งเทียบกับทุกคน
5. **Backpressure** — เกิดอะไรขึ้นเมื่อ client ช้าตามไม่ทัน และจะปกป้อง memory ของ server อย่างไร

## server ที่รันได้จริงตัวแรก

เดโมในโมดูลนี้คือ **Node server จริง** ที่สร้างบนไลบรารี [`ws`](/websocket-design-from-zero-to-hero/th/the-client-api/) — ไลบรารี WebSocket ที่ใช้กันแพร่หลายที่สุดสำหรับ Node เปิดอันใดก็ได้ใน **StackBlitz** เพื่อรัน `WebSocketServer` ของจริง จากนั้น `npm run client` ใน terminal ที่สองเพื่อเชื่อมต่อ

ตลอดทั้งโมดูลนี้ โค้ดที่คุณเห็นคือ **body ของ connection handler** — มันรันหนึ่งครั้งต่อ client ที่เชื่อมต่อ โดยมีสาม identifier อยู่ใน scope แล้ว: `ws` (socket ของ client รายนี้), `wss` (server, ใช้สำหรับ broadcast), และ `req` (upgrade request) ตัวอย่างด้านล่างทักทาย client ใหม่แต่ละราย, echo สิ่งที่พวกเขาส่งมา, และ log เมื่อพวกเขาจากไป

export const helloServerCode = `// Runs once per connected client. ws/wss/req are in scope.
ws.send('welcome — you are connected to a real ws server');

ws.on('message', (data) => {
  const text = data.toString();
  console.log('server received:', text);
  ws.send(\`echo: \${text}\`);
});

ws.on('close', () => console.log('a client disconnected'));`;

<NodeRunner node client:visible code={helloServerCode} />

เมื่อคุณเชื่อมต่อ client เข้ามา server จะส่งคำทักทายทันที จากนั้น echo ข้อความของคุณกลับมาพร้อม prefix `echo:` การ `ws.send` ตอน connect — ก่อนที่ client จะพูดอะไรเลย — คือส่วนที่ HTTP ไม่มีวันทำได้: server พูดก่อน ทุกอย่างที่เหลือในโมดูลนี้สร้างขึ้นบนการ holding connection นั้นให้เปิดค้างไว้และตัดสินใจว่าจะส่งอะไรผ่านมัน

<Callout title="งานสี่อย่างของ server">
WebSocket server **รับ upgrade**, **ติดตาม** ทุก connection ที่เปิดอยู่, **route** ข้อความขาเข้า, และ **broadcast** ไปยัง client จำนวนมากพร้อมกัน สามในสี่อย่างนี้มีอยู่ก็เพราะ connection เปิดค้างไว้ — ความถาวรนั้นคือเหตุผลทั้งหมดที่ทำให้การออกแบบ server ต่างจาก HTTP
